What is Stalwart

Stalwart Mail Server is a modern, Rust-based mail server distributed as a single binary. It supports SMTP, IMAP, JMAP, and includes built-in spam filtering, DKIM, and a management API.

NorthRelay uses Stalwart in a split-role architecture with two separate instances:

InstanceRoleLocationPurpose
Outbound RelayQueue + routingSanctuary VPSAccepts SMTP submissions, routes to correct pool, manages queue
Inbound MailboxIMAP + JMAPMailbox VPSReceives inbound mail, user mailboxes, dashboard inbox

These are independent Stalwart instances with separate configurations, separate admin tokens, and separate Management APIs.

Split-Role Architecture

Outbound Relay (Sanctuary)

The relay instance handles all outbound email:

Key characteristics:

  • Runs on the Sanctuary VPS alongside the dashboard
  • Uses host networking in Docker for SMTP port binding
  • Routes emails to shared or isolated relay based on user tier
  • DKIM signing is disabled (handled by OpenDKIM on relay VPSes)
  • Management API on port 8080 (internal only)

Inbound Mailbox

The mailbox instance handles all inbound email and user inbox access:

Key characteristics:

  • Runs on a dedicated Mailbox VPS
  • MX records point here (mx.northrelay.ca)
  • Provides IMAP and JMAP access for dashboard inbox
  • Uses SQL directory queries against PostgreSQL for authentication
  • Each user's inbox email = their subdomain slot email

IP Blocking Prevention

This is the most common operational issue with Stalwart.

Stalwart has an auto-ban feature that blocks IP addresses after failed authentication attempts. Blocked IPs are stored in RocksDB (not the TOML config), so they persist across restarts.

Symptoms

  • SMTP connection closes immediately without a banner
  • Telnet to port 25 connects then disconnects instantly
  • No error in application logs - the connection is dropped at the Stalwart level

Diagnosis

Fix: Unblock an IP

SQL Directory Queries

The mailbox Stalwart instance uses SQL queries against PostgreSQL for user authentication and mail routing. These queries are stored in RocksDB and queried at runtime.

Query Types

QueryPurposeSupports Catch-All
recipientsDetermine if an address should be acceptedYes
nameResolve a user account from an addressYes
emailsGet all allowed sender addresses for a userYes
secretsGet password hash for authenticationNo
membersGroup membership lookupNo

Managing SQL Queries

Catch-All Support

The recipients and name queries include catch-all logic: if no exact match is found, they check whether the domain has is_inbox_hosted = true in the domains table. If so, all mail to that domain routes to the domain owner.

Two-Pool Routing

The outbound relay routes emails to different relay VPSes based on the user's pool assignment:

User TierPool TypeRouteRelay VPS
Sandbox, MicroSharedWireGuard 10.0.1.2:2576.13.26.53
Startup, Scale, EnterpriseIsolatedWireGuard 10.0.2.2:2576.13.27.101

Routing is enforced by HTTP headers injected by the API:

  • X-NorthRelay-Pool-Type: Shared or Isolated
  • X-NorthRelay-Pool-Tier: The user's plan tier

Critical: Pool mixing is never acceptable. If routing is ambiguous, always default to the Shared pool to protect Isolated IP reputation. See the Two-Pool System for more details.
