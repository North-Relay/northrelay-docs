Stalwart Two-Layer Configuration

Stalwart Mail Server uses two configuration storage layers. Understanding this is critical for operations.

Layer 1: TOML Config File (Static)

The TOML file is loaded at startup and defines the base configuration:

Changes to the TOML file require a restart or config reload to take effect.

Layer 2: RocksDB Database (Runtime)

Runtime state is stored in Stalwart's embedded RocksDB database. This includes:

  • Blocked IP addresses (from auto-ban)
  • Session counters
  • Some dynamic settings overridden at runtime

Warning: Database entries take precedence over TOML file entries for the same key. If a setting exists in both places, the database value wins. Use the Stalwart CLI to inspect and manage database configuration.

Database Configuration

PostgreSQL Connection

NorthRelay uses PostgreSQL 15 with Prisma as the ORM. The schema source of truth is prisma/schema.prisma.

Schema Management

Key Schema Conventions

  • All table names use snake_case with @@map directive
  • Key enums: plan_tier, pool_type, user_role, user_status, email_status
  • Every model requires a corresponding Prisma model in schema.prisma

Redis Configuration

Redis is used for:

  • Rate limiting - Per-user SMTP send rate enforcement
  • Caching - Session data, DNS lookup caching
  • Queues - Background job processing

If Redis is unavailable, NorthRelay falls back to in-memory caching with reduced functionality.

DKIM Configuration

NorthRelay uses OpenDKIM on relay VPSes for DKIM signing, not Stalwart's built-in DKIM.

Why Relay-Based Signing

  • Stalwart's DKIM is disabled (queue.outbound.sign.enable = false)
  • OpenDKIM is battle-tested for dynamic domain signing
  • Signing at the network edge means signatures are never broken by intermediate processing

Domain Separation

NorthRelay enforces strict domain separation between corporate and user traffic:

DomainPurposeUsage
northrelay.caCorporateDashboard, docs, admin email - never for user SMTP traffic
north-pipe.caSystemUser subdomains, HELO identity, return-path - all user email traffic

This separation protects the corporate domain reputation from user sending behavior.
